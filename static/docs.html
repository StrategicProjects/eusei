Documentação

A eusei traduz os Web Services SOAP do SEI em JSON limpo. Toda chamada é um GET autenticado por Bearer token.

base /eusei
Fluxo de uma requisição

Autenticação

Todas as rotas /v1/* exigem um token no cabeçalho. As rotas públicas (esta página, /health, /openapi.json) não.

Authorization: Bearer <seu-token>

Convenções

Protocolo na query

A barra do número (/) não sobrevive no path através do proxy. Sempre use ?protocolo=.

Sinalizadores

Filtros sin_retornar_* aceitam S/N (padrão S) para incluir/omitir blocos.

Nomes originais

O JSON preserva os nomes do SEI (ex.: ProcedimentoFormatado); xsi:nil vira null.

Anatomia da resposta

Sucesso devolve { ok, dados }. Abaixo, o formato de um processo (/v1/procedimento):

{
  "ok": true,
  "dados": {
    "IdProcedimento": "30272383",
    "ProcedimentoFormatado": "0011…/2022-49",
    "Especificacao": "TOMADA DE PREÇOS…",
    "TipoProcedimento": { "Nome": … },
    "AndamentoGeracao": { … },
    "UltimoAndamento": { … },
    "Assuntos": [ … ],
    "Interessados": [ … ]
  }
}

Cache

As respostas JSON de /v1/* são cacheadas em memória (read-only), com single-flight e serve-stale quando o SEI cai. Cada resposta traz headers para o cliente decidir.

X-Cache

HIT · MISS · STALE · PARTIAL. STALE = SEI fora do ar, servindo o último valor bom.

Cache-Control · Age

private, max-age=<ttl> + Age coerentes com o TTL (listas ~6h; processo ~30s).

Forçar fresco

Envie Cache-Control: no-cache (revalida) ou no-store (busca fresco sem persistir).

Faults de SOAP de "não encontrado" são lembrados por pouco tempo (negative caching). O endpoint SSE /v1/andamentos/stream é exceção: não carrega esses headers (é um fluxo de eventos), mas respeita no-cache/no-store. Estatísticas em /health.

Consultas

processos · documentos · publicações · blocos
GET /v1/procedimento

Consulta um processo pelo protocolo. Aceita os filtros sin_retornar_* (assuntos, interessados, observacoes, andamento_geracao, andamento_conclusao, ultimo_andamento, unidades_procedimento_aberto, procedimentos_relacionados, procedimentos_anexados).

protocoloobrigatório
Exemplo 
curl -H 'Authorization: Bearer TOKEN' \
  '…/eusei/v1/procedimento?protocolo=0011108545.000056/2022-49'
GET/v1/procedimentos

Lote de processos. Cada item retorna { protocolo, dados, erro }.

protocolosobrigatórioseparados por vírgula
GET/v1/documento · /v1/documentos

Consulta um documento (ou um lote, por ?protocolos=). Filtros sin_retornar_*: andamento_geracao, assinaturas, publicacao, campos.

protocoloobrigatório
GET/v1/publicacao

Consulta uma publicação. Informe ao menos um: id_publicacao, id_documento ou protocolo_documento.

GET/v1/bloco · /v1/procedimento-individual

/bloco?id= consulta um bloco. /procedimento-individual exige id_orgao_procedimento, id_tipo_procedimento, id_orgao_usuario e sigla_usuario.

Andamentos & processo

GET/v1/andamentos

Linha do tempo do processo, ordenada por data/hora. Sem filtros, usa as tarefas 1–200 (timeline completa). Filtros: tarefas, andamentos, tarefas_modulos (listas), sin_retornar_atributos, ordenar.

Em processos grandes a consulta é fatiada em lotes de tarefas, executados em paralelo e mesclados — evita o HTTP 504 do gateway do SEI. A resposta inclui um bloco resumo (lotes, registros, parciais); parciais: true sinaliza que algum lote falhou e a timeline pode estar incompleta.

protocoloobrigatório
GET/v1/andamentos/stream · SSE

Mesma consulta de /v1/andamentos, em text/event-stream: emite um evento progresso a cada lote concluído (lotes_concluidos, total_lotes, registros_ate_agora) e, ao final, concluido com o payload completo. Para exibir o avanço ao usuário em processos grandes.

protocoloobrigatório
GET/v1/documentos-processo · /v1/publicacoes-processo

Documentos do processo (extraídos da timeline) e suas publicações. O SEI não expõe isso nativamente — são heurísticas. publicacoes-processo faz uma consulta por documento (paralelizadas) e pode levar alguns segundos.

protocoloobrigatório

SIP · Permissões

GET/v1/permissao

Lista permissões no Sistema de Permissões (SIP). Usa endpoint e credenciais próprios — requer SEI_SIP_CHAVE_ACESSO e SEI_SIP_ID_SISTEMA no servidor; sem isso, responde 400 com mensagem clara.

Erros

Falhas vêm como { ok:false, codigo, erro, detalhe } com status HTTP coerente. O codigo é estável para o cliente tratar.

parametro_invalido400falta parâmetro obrigatório
